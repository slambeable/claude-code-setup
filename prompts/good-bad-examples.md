# Хорошие и плохие промпты: примеры

Пять правил хорошего промпта:

1. **Одна задача** на промпт
2. **Укажи где** — файл, функция, строка
3. **Дай контекст** — почему это нужно
4. **Покажи паттерн** — ссылка на похожий код
5. **Критерий готовности** — билд, тесты, конкретная команда

Ниже — разбор на нескольких задачах. Пример 1 — из реального проекта, остальные — типовые задачи для иллюстрации.

---

## Пример 1. Валидация паттернов при экспорте

### ❌ Плохо

```
При экспорте JSON нужно проверять паттерны на ошибки.
И ещё покажи предупреждения прямо в списке, чтобы
было видно какие паттерны проблемные.
```

### ✅ Хорошо

```
Несколько раз экспортировал main.json и не заметил,
что у одного из паттернов data[] состоял из нулей —
игра спавнила пустой уровень.

Добавь валидацию в exportJson() в
src/features/level-editor/composables/useEditor.ts.

Перед Blob пройдись по паттернам: is_active &&
data[] только нули → ошибка. Покажи confirm()
со списком ID. За образец: syncAutoFields().

Убедись, что npm run build проходит.
```

## Пример 2. Добавить таймаут на API-запрос

### ❌ Плохо

```
Сделай таймауты на запросы к бэку, а то иногда висит
```

### ✅ Хорошо

```
В src/api/client.ts есть функция apiRequest — обёртка
над fetch. Добавь в неё таймаут 10 секунд через
AbortController. Если таймаут сработал — выброси
ApiError из src/api/errors.ts с кодом TIMEOUT.

За образцом поведения ошибок смотри обработку 503 в
этой же функции. Проверь, что существующие тесты в
src/api/__tests__/client.spec.ts не сломались.
```

## Пример 3. Рефакторинг дубликатного кода

### ❌ Плохо

```
В компонентах списков много одинакового кода,
вынеси куда-нибудь
```

### ✅ Хорошо

```
Посмотри src/views/UsersList.vue, src/views/TeamsList.vue
и src/views/ProjectsList.vue. У них одинаковая логика:
загрузка, состояние loading/error/empty, пагинация.

Вынеси общую логику в composable useList<T>() в
src/composables/useList.ts. Сигнатуру подбери сам,
но как минимум: items, loading, error, fetch(), hasMore.

За образцом структуры composable: useAuth в том же
каталоге. Убедись, что три переписанных view-компонента
собираются через npm run build и их e2e-тесты проходят.
```

## Пример 4. Дебаг сломанного теста (+ Plan Mode)

Этот пример скорее про комбинацию хорошего промпта с Plan Mode — для запутанных багов это часто полезнее, чем сразу просить правку.

### ❌ Плохо

```
Почему тест не работает?
```

### ✅ Хорошо

```
Тест src/features/editor/__tests__/save.spec.ts
падает на строке 42 с "Cannot read property 'id' of undefined".

Раньше работал. Последнее, что менялось — добавлял
новое поле version в Pattern в types/pattern.ts
и в фабрику createPattern в src/utils/factory.ts.

Прежде чем чинить — прочитай оба файла и spec,
скажи, в чём причина. Правки — только после
подтверждения от меня.
```
