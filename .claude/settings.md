# settings.json — разбор правил

Файл `.claude/settings.json` управляет разрешениями Claude Code: какие команды и операции выполняются автоматически, какие требуют подтверждения, какие полностью блокируются.

## Как работает система разрешений

Три списка: `allow`, `ask`, `deny`. Правила проверяются в порядке **deny → ask → allow**, первое совпадение выигрывает.

- `allow` — команда выполняется без подтверждения
- `ask` — каждый раз запрашивает подтверждение у пользователя
- `deny` — команда блокируется полностью
- Всё, что не попало ни в один список — поведение по умолчанию (обычно спрашивает)

## ⚠️ Важные ограничения

**Это defense in depth, не абсолютная защита.**

- Есть открытые баги в реализации: deny-правила иногда игнорируются (публично задокументировано, воспроизводимо). Особенно для чтения файлов вроде `.env`.
- Паттерны проверяют строку команды *до* выполнения, но не контролируют, что происходит внутри. `Bash(npm run *)` разрешает `npm run dev`, но если в `package.json` скрипт `dev` делает `rm -rf /` — `deny` этому не помешает.
- Bash-паттерны чувствительны к пробелам и флагам. Для force-push мы добавили четыре разных варианта паттерна, потому что `git push --force` и `git push -f` имеют разный порядок флагов в реальности.
- Для действительно важных вещей (ключи, credentials, prod БД) используйте OS-level защиту: разные пользователи, сендбокс, отдельные окружения. Не полагайтесь только на `settings.json`.

## Разбор по блокам

### allow: выполняется без подтверждения

Сюда попадает рутина, которая не может навредить:

- `npm run dev/build/test/lint/format/type-check` — стандартные скрипты проекта
- `npx vitest *`, `npx vue-tsc *` — запуск тестов и проверка типов
- Безопасные git-команды: `status`, `diff`, `log`, `add`, `checkout`, `switch`, `branch`
- `Read`, `Write`, `Edit`, `MultiEdit` — операции с файлами (кроме тех, что явно в `deny`)

**Принцип:** команда в `allow` только если я уверен, что её случайное выполнение не причинит вреда.

### ask: запрашивает подтверждение

Эти команды легитимны, но последствия серьёзнее — стоит глазами проверить каждый раз:

- `npm install *` / `npm uninstall *` — изменение зависимостей. Агент может предложить установить вредный пакет (опечатка имени, typosquatting).
- `git commit *` / `git merge *` — изменяют историю. Хочешь сам прочитать сообщение и подтвердить.
- `curl *` / `wget *` — сетевые запросы. Легитимны для локальной разработки, но теоретически могут утечь данные. Пусть каждый раз спрашивает.

### deny: блокируется полностью

Операции, которые практически всегда — ошибка:

- `rm -rf *`, `rm -fr *` — массовое удаление. Если реально нужно — можно сделать руками в терминале вне сессии Claude.
- `git push * --force`, `git push --force *`, `-f` варианты — force push в shared-ветку переписывает историю команды. Четыре паттерна, потому что Bash-правила не учитывают порядок флагов.
- `git reset --hard *` — потеря локальных изменений без возможности восстановить.
- `npm publish *` — случайная публикация внутренних пакетов.
- `Read/Write ./.env`, `./.env.*` — секреты окружения. **Помни про баги в реализации deny: на всякий случай не храни в `.env` ничего критичного.**
- `Read **/*.pem`, `**/*.key` — приватные ключи.

## Синтаксис паттернов

Формат: `ToolName(specifier)`.

- `Bash(command)` — точное совпадение одной команды
- `Bash(git status)` — ровно `git status`, без аргументов
- `Bash(git *)` — любая git-команда (звёздочка после пробела = word boundary)
- `Bash(git *)` **не матчит** `gitk` (из-за word boundary)
- `Bash(git*)` **матчит** и `git`, и `gitk` (без пробела — нет границы)
- `Bash(npm run *)` — `npm run dev`, `npm run test` и т.д.
- `Read(./path)` — чтение конкретного пути
- `Read(./.env.*)` — любой файл, начинающийся с `.env.`
- `Read(**/*.pem)` — любой `.pem`-файл в любой глубине проекта

Важно: **не полагайтесь на Bash-паттерны для ограничения аргументов**. Например, `Bash(curl https://api.trusted.com/*)` не защитит от `curl https://evil.com` — это документированное ограничение. Для URL используйте `WebFetch` правила.

## Расширение под свой проект

Что стоит добавить самому:

- Свои `npm run *` скрипты, специфичные для проекта
- Если используешь pnpm/yarn — скопируй блок `Bash(npm *)` под свой пакетный менеджер
- Если в проекте есть кастомные CLI (`./scripts/deploy.sh`) — добавь их в `ask` или `deny`
- Пути к конфигам, которые не должен трогать агент — в `deny` через `Write(./config/**)`

## Управление через UI

Команда `/permissions` внутри Claude Code показывает текущие правила и из какого файла они пришли (проект / пользователь / managed). Удобно для отладки, когда правило «не срабатывает» — часто дело в том, что более приоритетный scope его переопределил.
